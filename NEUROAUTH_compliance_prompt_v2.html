Prompt — Arquitetura de Compliance por Operadora

Versão melhorada do prompt original. Contexto de código preciso, critérios de aceitação objetivos, lógica de prioridade entre regras globais e overrides, e arquitetura das 7 abas com colunas exatas.

7 abas estruturadas 5 operadoras Planilha Excel entregue Cowork-ready
⚡ O que foi melhorado em relação ao prompt original
+
Lógica de prioridade explícita entre as 3 camadas: o prompt original não definia a ordem de consulta do motor. Agora está especificado: 03_REGRAS (global ANS) → 03_REGRAS (override operadora) → 04_DOCUMENTOS. Sem esta hierarquia o motor pode aplicar regras na ordem errada.
+
Critério objetivo de conclusão em cada aba: para cada aba, o prompt agora define o que "aba pronta" significa — não "preencher exemplos", mas "motor consegue validar campo X da operadora Y usando apenas esta aba".
+
Separação entre bloqueia_envio e gera_alerta especificada por regra: o original apenas definia as colunas. Agora o prompt define a lógica: regras globais ANS sempre bloqueiam; overrides de portal sempre geram alerta sem bloquear.
+
Compatibilidade com Planilha-Mãe v3 existente: o prompt original não mencionava as abas já existentes (PROC001–PROC040, MAT001+, etc.). A versão melhorada especifica que as 7 novas abas são adicionais — não substituem o que já existe.
≠
Aba 03_REGRAS reescrita como matriz única: o original sugeria uma aba por operadora. A versão correta é uma única aba com coluna operadora_id — mais fácil de consultar por filtro e menos manutenção futura.
≠
07_MAPEAMENTO é a aba crítica para o Make.com: o original a listou por último como se fosse auxiliar. Na arquitetura correta, é a primeira que o Make consulta para montar o payload — reposicionada na hierarquia.
−
Removida sugestão de "aba por operadora": criaria 5–7 abas duplicadas com a mesma estrutura. Substituída por coluna operadora_id com filtro — mais limpo, mais escalável, menos erro de manutenção.
// arquitetura — 7 abas
01_OPERADORAS
Cadastro de Operadoras
Metadados: ANS, canal, portal, prazos, tipo de guia, CBOs aceitos, tabela de códigos.
Motor consulta: ao identificar o convênio do paciente
02_CAMPOS_TISS
Campos Base TISS
Todos os campos numerados da guia padrão TISS. Referência canônica de obrigatoriedade base.
Motor consulta: para saber quais campos são obrigatórios em qualquer convênio
03_REGRAS
Matriz de Compliance — ABA PRINCIPAL
Regras globais ANS + overrides por operadora. Uma linha por regra, coluna operadora_id. Inclui bloqueia_envio, gera_alerta, fonte normativa.
Motor consulta: PRIMEIRO — antes de qualquer validação
04_DOCUMENTOS
Documentos Exigidos
Checklist por operadora e cenário. Define o dossiê a montar antes de submeter autorização.
Motor consulta: SEGUNDO — para gerar checklist da secretária
05_OPME
Regras e Materiais OPME
Regras de OPME por operadora + tabela de materiais neurocirurgia com códigos TUSS e marcas comuns.
Motor consulta: quando necessita_opme = Sim
06_GLOSA
Base Anti-Glosa
Top 17 motivos de glosa por operadora com causa-raiz, campo relacionado e ação preventiva.
Motor consulta: TERCEIRO — para alertas preventivos finais
07_MAPEAMENTO
Mapeamento NEUROAUTH ↔ TISS
Ponte entre cada campo do index.html e o campo TISS correspondente. Funda o payload JSON enviado ao Make.
Make consulta: ao montar o payload e validar estrutura JSON
// lógica de prioridade — ordem de consulta do motor
1°
03_REGRAS — regras GLOBAL (operadora_id = "GLOBAL")
Regras da ANS. Aplicam-se a todos. Sempre bloqueiam se bloqueia_envio = SIM. Nenhuma operadora pode sobrepor.
SEMPRE
2°
03_REGRAS — regras da operadora específica (operadora_id = "UNI" | "BRA" | "SUL" | "HAP" | "AMI")
Overrides da operadora. Podem bloquear ou apenas alertar. Aplicam-se somente ao convênio selecionado.
SE CONVÊNIO
3°
04_DOCUMENTOS — checklist do cenário
Filtrar por operadora + cenário (cirurgia coluna, OPME, neurocirurgia). Gerar lista de documentos que a secretária precisa anexar.
SEMPRE
4°
05_OPME — validações OPME
Executar somente se necessita_opme = Sim. Verificar: 3 marcas, ANVISA, código TUSS, justificativa individual.
SE OPME
5°
06_GLOSA — alertas preventivos finais
Cruzar campos do formulário com top motivos de glosa. Gerar alertas visuais antes do envio. Não bloqueia — apenas avisa.
ALERTA
✓
Envio liberado → 07_MAPEAMENTO constrói o payload JSON
Traduz campos do formulário para estrutura TISS + campos extras do Make. Este é o contrato entre frontend e automação.
